# Feature Intake Protocol

Feature intake 的目标是**充分理解需求并从多源信息综合建议**。不要跳过需求确认直接写代码。

## 三源信息模型

模型在 intake 阶段必须从三个信息源并行采集，每个源的权重均等：

### 1. 知识库（项目内部）

扫描范围：
- 项目文档（README、docs/、spec 文件）
- 已有代码模式和架构约束
- 历史 ADR 决策及其后果
- 技术栈版本和兼容性限制
- 已有的相关功能（避免重复建设）

行动：
- 阅读相关模块代码，理解现有架构
- 检查是否有类似功能或未完成的相关工作
- 识别技术栈约束（依赖版本、API 兼容性等）

### 2. 外部信息

采集渠道：
- `WebSearch` 搜索相关技术方案和最佳实践
- API / SDK 官方文档
- 设计参考和竞品分析
- 社区讨论和已知 pitfall
- 相关 RFC / 标准文档

行动：
- 针对核心技术点进行 WebSearch
- 查阅涉及的第三方 API/SDK 文档
- 如有 UI 需求，搜索设计参考和 UX 最佳实践
- 汇总外部信息中对方案选择有影响的关键发现

### 3. 用户输入

必须确认的信息：
- **背景**：为什么要做这个功能？
- **目标**：解决什么问题？给谁用？
- **范围**：做什么 + **不做什么**（显式排除同样重要）
- **验收标准**：什么情况下算完成？
- **约束和偏好**：技术偏好、时间限制、风格要求
- **依赖项**：是否依赖其他功能或外部服务
- **风险点**：用户已知的潜在问题

行动：
- 通过对话逐项确认（一次一个问题，优先多选题）
- 不要假设用户意图，显式确认模糊点
- 将用户的回答结构化记录

## 综合建议流程

```
并行采集三源信息
  ↓
识别信息之间的冲突或互补关系
  ↓
形成需求摘要：
  - 功能目标
  - 关键约束（来自知识库）
  - 推荐做法（来自外部信息）
  - 用户期望（来自用户输入）
  ↓
呈现给用户确认
  ↓
用户确认？
  ├─ 是 → 进入 ADR Decision
  └─ 否 → 补充采集 → 重新综合
```

## 准入检查

进入 ADR Decision 前必须满足：

- [ ] 需求目标明确且用户已书面确认
- [ ] 验收标准已定义（至少 2 条可验证的标准）
- [ ] 范围已界定（做什么 + 显式排除不做什么）
- [ ] 关键依赖已识别且可获取
- [ ] 已知风险已记录

## 输出格式

intake 完成后，在 issue 状态中记录：

```json
{
  "stage": "ADR",
  "phase": "spec",
  "intake_summary": {
    "goal": "功能目标描述",
    "scope_in": ["做什么1", "做什么2"],
    "scope_out": ["不做什么1", "不做什么2"],
    "acceptance_criteria": ["标准1", "标准2"],
    "constraints": ["约束1", "约束2"],
    "dependencies": ["依赖1"],
    "risks": ["风险1"],
    "external_findings": ["外部信息关键发现1"]
  }
}
```

## 与 ADR Decision 的衔接

intake 完成后，进入 ADR Decision 阶段时：

1. 将 intake 摘要作为 ADR 的 context 基础
2. 使用 superpowers `brainstorming` skill 驱动方案设计
3. brainstorming 过程中可以随时引用 intake 阶段收集的三源信息
